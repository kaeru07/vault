# 1日分情報収集テスト計画（3 情報源・Phase 1 検証）

> Issue #32。完全自動化（#25/#28）の前に、**3 情報源で 1 日分の収集 → 保存 → 案生成 → 上位 5 件抽出を 1 回だけ手動検証**する計画。

> Claude Code はテスト計画 + 詰まり想定 + 自動化時の注意点を**事前整理**し、実取得は人間判断＋承認後に実施。

## 1. 対象情報源（Issue 本文 + #29 Phase 1 推奨に基づく差し替え）

| Issue #32 本文 | 本計画での採用 | 差し替え理由 |
|---|---|---|
| Google Trends | **採用**（公開 Daily Trends RSS） | RSS で取得可・規約緩め |
| Google Play ランキング | **採用**（HTML スクレイプ・規約確認後） | 既存 scrape-lab-v2 流用候補 |
| YouTube 検索 / 急上昇 | **採用**（YouTube Data API v3・無料枠） | 公式 API・無料枠あり・キー管理ルール必要 |

> 仮説: #29 では Reddit / iTunes Search / RSS 集約を Phase 1 推奨としたが、Issue #32 本文は GTrends / GPlay / YouTube を指定。**Issue 本文を尊重**し、この 3 源で計画作成。実装着手時に Phase 1 候補との優先順位を最終確認。

## 2. 取得手順（テスト・1 日 1 回）

### Step A. Google Trends（Daily Trends RSS）
- URL（例）: `https://trends.google.co.jp/trends/trendingsearches/daily/rss?geo=JP`
- 取得: curl + RSS パーサ。User-Agent 明示
- 保存: `06_research/daily/2026-MM-DD/google-trends.ndjson`（NDJSON / #30 規約準拠）
- 想定件数: 10〜20 件 / 日

### Step B. Google Play（HTML スクレイプ・規約確認後）
- 対象: トップ無料カテゴリ（カードゲーム / アクション 等のニッチ × 興味カテゴリ）
- 取得: Playwright / Cheerio（既存 note アプリ実装が参考）
- レート: 1 req / 2〜5 秒目安。robots.txt 事前確認必須
- 保存: `06_research/daily/2026-MM-DD/play-ranking.ndjson`
- 想定件数: 50〜100 件 / 日（カテゴリ上位 10〜20 件 × 数カテゴリ）

### Step C. YouTube Data API v3（無料枠）
- 検索クエリ: 麻雀 / Shorts / トレンド系
- API キー: `.env.YOUTUBE_API_KEY`（[[../03_prompts/GitHub反映ルール]] 機密ガード必須・コード化時に .gitignore 確認）
- 無料枠: 10,000 ユニット/日。検索 100 ユニット/1 リクエストで 1 日約 100 検索可
- 保存: `06_research/daily/2026-MM-DD/youtube-search.ndjson`
- 想定件数: 50〜200 件 / 日

## 3. 案 30 件生成（手動 or 半自動）

`05_monetization/idea_pool/_README.md`（Issue #26）の生成ルールに従う:

- 入力: 上記 3 源の NDJSON 合算（重複除去後 dedupKey で 100〜200 件想定）
- 生成: トレンド × 収益化手段（[[../05_monetization/収益化手段台帳]]）× 動画導線 の組み合わせで 30 案
- 粗 score（[[../05_monetization/案プール自動昇格ルール]] と整合）
- 保存: `05_monetization/idea_pool/2026-MM-DD.ndjson`
- 出力: 30 件（idea status）

## 4. 上位 5 件抽出（Issue #27 ルール適用）

- sortKey 降順で上位 5 件選定（収益化インパクト中核重視）
- 安全弁: 根拠 ≤ 2 かつ 収益期待 ≤ 2 は除外
- 既存 candidate（candidate-001〜004）との核ワード突合 → 重複は昇格対象外
- 出力: `05_monetization/scenarios/candidate-<NNN>.md`（最大 5 件）

## 5. 詰まり想定（事前整理 = Issue #32「詰まった点と自動化時の注意点」）

| # | 想定詰まり | 対策案 |
|---|---|---|
| D1 | Google Trends RSS の地域指定や言語混在 | `geo=JP&hl=ja` 明示 / 結果が空ならフォールバック |
| D2 | Google Play HTML 構造変化（A/B テスト等） | セレクタ複数候補 / 失敗時は raw HTML を保管しデバッグ |
| D3 | Play robots.txt 規約変更 | 実装着手時の最新確認 + 取得不可なら除外 |
| D4 | YouTube API キー漏洩リスク | .env + .gitignore + Vault 機密スキャン 3 重 |
| D5 | YouTube API 無料枠超過 | 検索数を抑制 / 1 日 50 検索以下に制限 |
| D6 | dedupKey の Unicode 正規化漏れ（NFKC） | #30 規約準拠 + サンプル検証 |
| D7 | 30 案生成のテンプレ過学習（似た案ばかり） | 手段 × トレンド × 動画導線の 3 軸を強制 |
| D8 | 上位 5 件が全部既存 candidate と重複 | 重複時はランクダウンで補充 |
| D9 | 実取得失敗時の部分実行 | 1 源失敗で他源は続行・失敗ログは progress へ POST |
| D10 | 機密混入（取得 raw に個人情報） | 保存前に機密スキャン（#30 規約） |
