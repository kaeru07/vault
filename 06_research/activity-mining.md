# activity-mining（作業履歴からの抽出）

> Git 履歴 / Claude Code 作業履歴 / progress アプリのデータから、**興味・未完了・収益化候補・再開候補を抽出する**ための方針メモ。
> まだ実装はない。将来やる前提でアイデアを置いておく場所。

---

## 目的

- 自分の**「気付かないうちに繰り返している作業」**を見つける
- **手元の資産**（既に書いたコード・ノート）の中から収益化できそうなものを抽出する
- 放置案件・再開候補を**忘れる前に**拾い上げる
- 興味の波を可視化して、深掘り価値のあるテーマを定量的に判断する

---

## 何を抽出したいか

### 興味分析

- 同じテーマで何度も commit / 作業しているもの
- inbox / 未整理メモ / today にしばしば出てくるキーワード
- 作業時間が偏っているアプリ・領域
- 朝会で繰り返し言及される話題

### 放置案件

- 最終 commit から N ヶ月触っていないアプリ
- progress アプリで `in_progress` のまま N 週間経過しているタスク
- ToDo に書いたまま消化されていない項目
- 棚上げ済みだが「あれどうなったっけ」が定期的に浮上するもの

### 収益化候補

- 公開可能な完成度に近いのに公開していないアプリ
- 同じ問題を別アプリで解決している重複領域（統合候補）
- 自家用に作ったツールで他人にも価値がありそうなもの（progress テンプレ販売など）
- ノートに繰り返し出てくる「困っている」「面倒」（= 課題 = 市場）

### 再開候補

- 放置中だが**外部状況が変わった**もの（競合の動き / プラットフォーム変更 / 自分のスキル向上）
- 一度棚上げしたが、新しい知見と組み合わせると勝てそうなもの
- 中途半端な実装が残っており「**もう少し**」で公開できるもの

---

## データソース

| ソース | 使い方 |
|---|---|
| `git log` / `git shortlog` | アプリ別の最終 commit / 頻度 |
| Claude Code 会話履歴（要保存設計） | 言及キーワードの頻度 |
| progress アプリの `work-log.ndjson` | 作業時間 / 件数 / アプリ別配分 |
| `engineering/reports/focus-work-*.md` | 危険作業 / 中断ポイント |
| Obsidian Vault | inbox / today / app ノートの追記頻度 |
| 各 SaaS（YouTube / Play Console / X） | 公開後の反応データ |

---

## 想定アウトプット

### 1. 月次レポート（自動生成）

- アプリ別作業時間
- commit 頻度の Top 5
- inbox に新規追加されたが処理されていないメモ
- progress 上で `in_progress` のまま 2 週間経過したタスク
- → Obsidian の `06_research/activity-YYYYMM.md` に保存

### 2. アラート

- 「最終 commit から 3 ヶ月経過」したアプリのリスト
- 棚上げ済み案件で「再評価条件」が満たされたもの
- progress の `review_pending` が滞留しているもの

### 3. 収益化シグナル

- 「公開ほぼ完了」状態のアプリで未公開のもの
- 同じテーマを 3 件以上のノートで言及している = 興味の濃いテーマ

## 段階的に作る

| フェーズ | 内容 |
|---|---|
| Phase 0 | 手動でこのページを読み返して気付きを残す（**今ここ**） |
| Phase 1 | progress アプリの集計ビューを 1 つ追加（アプリ別作業時間） |
| Phase 2 | git log と progress を突き合わせる CLI を 1 本 |
| Phase 3 | Obsidian Vault のキーワード頻度抽出 |
| Phase 4 | 自動レポート → 月次で Obsidian に保存 |

実装は急がない。**思いついた仕組みをまずここに書く**のがこのページの役割。

---

## 注意点

- 「効率化のための効率化」にしない。**収益・公開・学習に直結するシグナル**だけを残す
- 自動化に時間を使うより、まず**月 1 回手動で振り返る**ところから始める
- データの粒度を細かくしすぎない（数値より「気付き」を取りに行く）
