# 旧Vault運用と現ゴール運用の統合点検（2026-06-20）

> 目的: 旧Vault側の運用（candidate / vloop / Epic A–D / ChatGPT承認）と、現progressアプリのゴール運用（Goal / ゴール承認 / 自動実行 / ExecutionRun）が**並行2系統**になっている状態を点検し、最新のゴール運用へ統合する段階別計画を出す。

## 1. 結論

「候補 → 承認 → キュー → 実行 → 完了ログ → 進捗」という**同じ運用ループが2つ**存在する。
- 旧: Vault（`05_monetization/` + `20_reviews/` + vloopスキル + GitHub Issue + ChatGPT承認）。最終更新は概ね 2026-05-25（vloop11）で**停滞気味**。
- 新: progress アプリ（Goal / ゴール承認 / auto-queue / factory-runner / ExecutionRun）。**現役・自動実行まで回る**。

→ progress 側を正本に寄せ、Vault 側は「判断ログ・ナレッジ・ChatGPTレビュー入口」に役割を絞るのが統合方針。

## 3. 統合方針（正本の寄せ先）

- **実行・キュー・完了履歴・ToDo・Goal進捗 = progress を正本**（自動実行まで回る方が強い）。
- **Vault = 判断ログ・ナレッジ・ChatGPTレビュー入口**に役割を限定（`20_reviews/_review_queue.md` は維持）。
- **収益化候補（candidate）= progress Goal へ移行**し、Vaultシナリオ詳細はリンク参照に。
- **vloop = 廃止 or progress連携に再定義**（Epic完了優先はfactory-runnerが担えるか検証）。

## 4. 段階別統合計画（→ ゴール承認に登録）

- **P1 棚卸し・対応表の確定**（低リスク・本ファイルが起点）
- **P2 収益化候補(candidate)をprogress Goalへ移行**（状態引き継ぎ・Vaultはリンク化）
- **P3 ToDo/キューの一本化**（vloop_queue/案件別ToDo → GoalTodo/auto-queue）
- **P4 承認フローの一本化**（ChatGPT承認待ち と ゴール承認/review_queue の役割確定）
- **P5 vloop運用の位置づけ確定**（廃止 or progress連携へ再定義）
- **P6 旧ドキュメントのアーカイブと索引更新**（移行済みページをarchive、関係図/Vaultの見方を現行化）

## 5. 注意・リスク

- 破壊的操作（Vaultフォルダ削除・大量リネーム）はユーザー判断（CLAUDE.local バックアップ方針）。各Pは「移行→リンク化→アーカイブ」の順で非破壊的に。
- 収益化候補の公開可否・撤退判断は承認必須。
- 二系統が一時的に併存する移行期は、どちらが正本かを各ファイル冒頭に明記して混乱を防ぐ。
