## 1. 既存資産の現状

### 既存「Vault の見方ガイド」（[[../00_inbox/Vaultの見方_どこを見れば何がわかるか]]）

- 場所: `00_inbox/Vaultの見方_どこを見れば何がわかるか.md`
- 性格: 個人運用ノウハウのまとめ（Vault 全体の探し方・読み方）
- 内容: Vault 構造 / 検索パターン / iPhone Obsidian 操作 / トラブル対応
- 利用者: ユーザー（あなた）
- フォーマット: Markdown（iPhone Obsidian で読む前提）

### candidate-007 の MVP（[[../90_prototypes/vault-search-cheatsheet/README]]）

- 場所: `90_prototypes/vault-search-cheatsheet/`
- 性格: 公開可能な静的 HTML（検索パターン早見表）
- 内容: 検索ボックス + 20 件キーワード対応表 + iPhone/GitHub Tips + トラブル対応フロー
- 利用者: Vault / Obsidian ユーザー層（一般）
- フォーマット: HTML（外部 CDN ゼロ・file:// 動作）

### 重複範囲

| 重複項目 | 既存ガイド | candidate-007 |
|---|---|---|
| Vault の構造説明 | あり | なし（HTML 簡略版）|
| 検索キーワード一覧 | テキスト形式 | 検索ボックス + テーブル |
| iPhone Obsidian 操作 | テキスト | スクショ予定 |
| トラブル対応 | テキスト | フロー図 |

→ **検索キーワード一覧と iPhone Obsidian 操作で重複**。Vault 構造説明は既存ガイドの独自範囲。

## 2. 統合方針 A / B / C の比較

### 案 A: 既存ガイドを残し、本候補は HTML Web 版として併存

```
[既存ガイド (Markdown)]  並走  [Vault Search Cheatsheet (HTML)]
       ↑                          ↑
   iPhone Obsidian              一般ユーザー
```

#### メリット
- 既存ガイド利用者（あなた）の習慣変更ゼロ
- HTML 版独自の検索ボックス UX が活きる
- 公開リスクが低い（既存ガイドは個人運用のまま）

#### デメリット
- **重複保守 2 重**（検索キーワード追加時に両方更新）
- 検索キーワードが片方しか更新されないリスク
- 「同じ目的のページが 2 つある」状態が継続

#### 想定運用
- 既存ガイド: あなたの個人運用記録（更新は気が向いたとき）
- candidate-007: 一般公開向け（検索キーワード追加運用は週 1 件）

### 案 B: 既存ガイドを索引化、本候補が実体ページ ★ Claude 推奨

```
[既存ガイド (Markdown / 索引化)]  →  [Vault Search Cheatsheet (HTML)]
       ↑                                    ↑
   iPhone Obsidian                      一般ユーザー
   （リンクで HTML へ飛ぶ）
```

#### メリット
- **重複保守ゼロ**（実体は 1 箇所）
- 検索キーワードは HTML 側で一元管理
- 既存ガイドは「個人 Vault 運用ノウハウの索引」に役割集約
- iPhone Obsidian からも GitHub Web で HTML を見られる（既存ガイドのリンクで誘導）

#### デメリット
- 既存ガイド利用者（あなた）の習慣変更が必要（リンク経由で見る）
- iPhone Obsidian で HTML はそのままレンダリングできない（GitHub Web で開く必要）
- 既存ガイドが空に近くなる（索引化）

#### 想定運用
- 既存ガイド: 「検索キーワード一覧 → HTML 版を見てください」のリンク + 個人 Vault 運用ノウハウ（HTML 化できない範囲）
- candidate-007: 一般公開向け実体ページ（検索キーワード追加運用は週 1 件）

### 案 C: 既存ガイドへ統合、本候補は HTML サンプルのみ残す

```
[既存ガイド (Markdown / 統合先)] ← 内容統合 ← [Vault Search Cheatsheet (HTML サンプル)]
       ↑
   iPhone Obsidian
```

#### メリット
- **重複保守ゼロ**（実体は既存ガイド側）
- 既存ガイド利用者の習慣変更最小
- iPhone Obsidian で常に最新を読める

#### デメリット
- HTML 独自の検索ボックス UX が失われる
- 「公開向け資産」がなくなる（収益化導線が弱る）
- HTML はサンプル止まり（更新運用が止まる）

#### 想定運用
- 既存ガイド: 検索キーワード一覧含めて完全統合
- candidate-007: HTML サンプル（更新運用なし・公開なし）

## 3. 比較表（一覧）

| 観点 | 案 A 併存 | 案 B 索引化（推奨）| 案 C 既存に統合 |
|---|---|---|---|
| 重複保守コスト | **高（2 重）** | 低（ゼロ）| 低（ゼロ）|
| 既存ガイド利用者への影響 | ゼロ | 中（習慣変更）| 最小 |
| HTML 独自 UX 維持 | あり | あり | なし |
| 公開向け資産 | あり | あり | なし |
| 収益化導線 | あり | あり | **なし** |
| iPhone Obsidian で最新を読める | 既存ガイド側のみ | × HTML は GitHub Web 必要 | ◯ 完全 |
| 更新運用の負担 | 高（2 重）| 低 | 低 |
| 役割明確化 | 弱い | **明確（索引 vs 実体）** | 統合（明確）|

## 4. Claude 推奨案 = B（既存ガイド索引化）の根拠

### 主な根拠

1. **重複保守コストがゼロ**: 検索キーワード追加運用が 1 箇所で済む（週 1 件 × 案 A の半分の時間）
2. **収益化導線を維持**: candidate-007 の `revenueImpact: low` を実現するために HTML 版の公開が必要
3. **役割明確化**: 既存ガイドは「個人運用ノウハウの索引」、HTML は「一般向け検索パターン早見表」と役割を分離
4. **iPhone Obsidian の制約は許容可能**: 既存ガイドからリンクで GitHub Web へ飛ぶ運用は、検索パターン参照頻度（週 1-2 回）であれば負担にならない

### 案 A を退ける理由

- 重複保守 2 重で **運用負担が想定の 2 倍**
- 検索キーワードが片方しか更新されない事故リスク
- 「同じ目的のページが 2 つある」状態は Vault 全体棚卸し（Issue #59）の方針と矛盾

### 案 C を退ける理由

- 公開向け資産がなくなり **収益化シナリオが破綻**（candidate-007 の趣旨と矛盾）
- HTML 独自の検索ボックス UX が失われる
- 個人運用ノウハウと一般向け早見表が同居して既存ガイドが冗長になる

## 5. 案 B 採用時の具体手順（approved + 統合方針 B 決定後の Day2 作業）

### 5-1. 既存ガイドの整理（Day2 / 1-2 時間）

1. 既存ガイドから「検索キーワード一覧」セクションを抽出（重複範囲）
2. 抽出した内容を HTML 版（candidate-007）の 20 件テーブルと差分マージ（HTML 側に統合）
3. 既存ガイドの該当セクションを「[[../90_prototypes/vault-search-cheatsheet/README]] を参照してください」のリンクに置き換え
4. 既存ガイドの「iPhone Obsidian 操作」も同様にリンク化（HTML 側にスクショ付きで集約）

### 5-2. 既存ガイドに残すもの

- Vault 構造の説明（HTML 化しにくい個人運用ノウハウ）
- iPhone 同期トラブル時の復旧手順（個別状況に依存）
- HTML 側へのリンク（インデックス役割）

### 5-3. HTML 側への追加

- 既存ガイドからマージした検索キーワード（重複削除済）
- 既存ガイドからマージした iPhone Tips（スクショ付き）

---

## 6. 案 B を選ばないケース（hold / reject の判断材料）

- 既存ガイドを「Vault 内のみで完結する個人ノウハウ集」として維持したい意向が強い場合 → 案 A 採用
- 公開する意向自体が弱い場合 → 案 C 採用 + candidate-007 を hold or reject
- 別の統合案（D / E など）が出てきた場合 → ChatGPT 提案を受けて再評価
