# LLM Chooser 判定基準客観化方針

> [[candidate-006]]（90_prototypes/llm-chooser/）の **B2 判定基準客観化**（N-03 既知制約）を解消するための方針 1 ページ。
> 完全な客観化は難しいが「個人観察を観察可能な指標で裏付ける」アプローチで承認進行可能にする。

> [!important] 目的
> candidate-006 の判定基準が個人観察ベースであるという既知制約（N-03）を「観察可能な指標で裏付ける」方針で **ChatGPT 方向性レビュー可能状態に到達**させる。完全な客観化は次サイクル以降の課題として残す。

> [!note] 用語注（Issue #69）
> - 判定基準客観化 = 「どの LLM が合うか」の判定根拠を観察可能な指標で裏付けること
> - LLM Chooser = 質問 4 種に答えて Claude / Codex / Gemini / ローカル LLM を提案する静的 HTML
> - MMLU / HumanEval = LLM 能力を測る代表的なベンチマーク
> - Artificial Analysis = LLM のレイテンシ / スループットを集計する公開サイト
> - 利用ログ = Claude Code / Codex / Gemini 等の自分の使用記録
> - 自己申告 = クイズ回答者が自ら入力する条件（用途・速度重視・予算 等）

---

## 1. 既知制約（N-03）の整理

candidate-006 の判定基準は以下に依存:

| 判定要素 | 現状の根拠 | 問題点 |
|---|---|---|
| 用途別のおすすめ LLM | 個人観察（Claude Code / Codex / Gemini / ローカル LLM 利用経験）| 観察者バイアス・少数事例 |
| 速度重視 vs 品質重視の振り分け | 主観評価 | 同じ LLM でも条件で変わる |
| プライバシー要件への対応 | 個人観察 | ユーザーの具体要件と一致しない |
| 月額予算と LLM 価格帯のマッチング | 個人観察 | 公開価格は変動・モデル更新で陳腐化 |

### N-03 制約を完全解消する場合のコスト

- 全 LLM の用途別性能を網羅的に再測定 → 非現実的（モデル更新が早すぎる）
- 大規模ユーザー調査（n=100+）→ 個人開発のスコープ外
- → **完全解消ではなく「裏付け方針」で承認進行可能化**を採用

---

## 2. 客観化アプローチ（3 層構造）

判定根拠を「個人観察」から「個人観察 + 観察可能な指標」へ段階的に昇格させる:

```
[層1] ユーザー自己申告（クイズで取得）
        ↓
[層2] 公開ベンチマーク（MMLU / HumanEval / Artificial Analysis）
        ↓
[層3] 個人観察 + 利用ログ（カード裏面に根拠として表示）
```

### 層 1: ユーザー自己申告（クイズで取得）

クイズで以下を取得 → 判定の前提条件にする（個人観察に依存しない）:

| 質問 | 取得項目 | 判定への寄与 |
|---|---|---|
| Q1 主要用途 | コード / 文章 / マルチモーダル / プライバシー重視 | 大分類の振り分け |
| Q2 速度 vs 品質 | 速度優先 / 品質優先 / バランス | モデルサイズ選択 |
| Q3 月額予算 | 〜$0 / $0-20 / $20-100 / $100+ | 有料 / 無料層振り分け |
| Q4 プライバシー要件 | クラウド可 / ローカルのみ | クラウド LLM 除外判定 |

→ **Q1-Q4 は完全に観察可能**（ユーザーが直接答える）

### 層 2: 公開ベンチマーク（外部一次データ）

層 1 の自己申告に応じて、層 2 の公開ベンチマークから対応 LLM を絞り込む:

| 用途 / 重視点 | 参照する公開ベンチマーク | データソース |
|---|---|---|
| コード生成 | HumanEval / SWE-bench | Papers with Code / 各 LLM 公式 |
| 一般推論 | MMLU / MMLU-Pro | Papers with Code |
| 速度（tokens/sec） | Artificial Analysis | artificialanalysis.ai |
| 価格（$/M token） | 各 LLM 公式 API ページ | Anthropic / OpenAI / Google 公式 |

→ **公開ベンチマーク自体が再現可能**（誰でも同じソースを参照できる）

注: スコアは LLM バージョンごとに変わるため、**取得日付** を必ず併記する（例: `MMLU 89.2 / 取得日 2026-05-25 / source: papers-with-code`）。

### 層 3: 個人観察 + 利用ログ（裏付け表示）

層 1+2 の結果に対して、**個人の実利用ログを「カード裏面の裏付け」として表示**:

```
推奨: Claude Sonnet 4.6
  ・公開ベンチ: HumanEval 92.0（取得日 2026-05-25）
  ・公開ベンチ: MMLU 87.8（取得日 2026-05-25）
  ・個人観察: コード生成タスクで応答品質が高い（Claude Code 利用 100+ セッション / 期間 3 ヶ月）
  ・利用ログ: 平均応答時間 8.5s / エラー率 <2% / 期間 2026-02 〜 2026-05
```

→ 個人観察は**裏付けとして残す**が、層 1+2 と矛盾する場合は層 1+2 を優先する旨を明記。

## 3. 実装時の具体反映（candidate-006 への落とし込み）

### 3-1. クイズ拡張（B5 連動）

現状のクイズ 4 種を以下のように整理:

| Q | 既存 | 拡張後 |
|---|---|---|
| Q1 | 用途 | 用途（コード / 文章 / マルチモーダル / プライバシー重視）|
| Q2 | 速度 vs 品質 | 速度 vs 品質 vs バランス（3 択化）|
| Q3 | 月額予算 | 月額予算 4 段階（〜$0 / $0-20 / $20-100 / $100+）|
| Q4 | プライバシー | クラウド可 / ローカルのみ |

### 3-2. カード裏面に「3 層根拠」を表示

各 LLM カードの裏面に以下を表示:

```
[公開ベンチ]
  ・MMLU: NN.N（取得日 YYYY-MM-DD）
  ・HumanEval: NN.N（取得日 YYYY-MM-DD）
  ・Artificial Analysis tokens/sec: NN.N（取得日 YYYY-MM-DD）

[個人観察 + 利用ログ]
  ・利用セッション数: N+
  ・期間: YYYY-MM 〜 YYYY-MM
  ・主な気づき: 1-2 行
```

### 3-3. 判定結果の表示形式

```
あなたへのおすすめ: {LLM 名}

[なぜそう判定したか]
1. あなたの回答: 主要用途={Q1} / 速度重視={Q2} / 月額予算={Q3} / プライバシー={Q4}
2. 公開ベンチ: 該当用途で上位スコア（MMLU={値} / HumanEval={値}）
3. 個人観察: {要約 1 行}

[次点候補]
- {LLM 名 2}（理由: {1 行}）
- {LLM 名 3}（理由: {1 行}）
```

→ 「**なぜそう判定したか**」を 3 行で説明することで観察可能な指標を可視化。

## 4. 維持運用ルール（B4 と連動）

公開ベンチマークは定期的に更新が必要:

| 更新頻度 | 対象 | 担当 |
|---|---|---|
| 月 1 回 | MMLU / HumanEval スコア | 手動更新（card 裏面の取得日付を更新）|
| 月 1 回 | Artificial Analysis tokens/sec | 手動更新（同上）|
| LLM バージョン更新時 | 価格情報 / 新モデル追加 | 手動更新（公式 API ページ確認）|
| 週 1 回 | 個人利用ログ 1-2 件追加 | 自然蓄積（B4 と連動）|

→ **完全自動化はしない**（公開ベンチマークの取得 API は提供元によって不安定 / 手動更新で十分）。

## 5. ChatGPT 方向性レビュー時の論点

| 論点 | デフォルト案 | ChatGPT に確認したいこと |
|---|---|---|
| 3 層構造の妥当性 | 採用 | 層 1+2+3 の優先順位（層 1 が最優先 / 層 2 で絞り込み / 層 3 で裏付け）が妥当か |
| クイズ拡張範囲 | Q1-Q4 を 4 → 5 択化（プライバシー追加）| 質問数を増やしすぎないか / UX を損なわないか |
| 個人観察の扱い | 裏付けとして残す（層 1+2 と矛盾時は層 1+2 優先）| 個人観察を完全に消すべきか / 残すべきか |
| 公開ベンチの更新頻度 | 月 1 回手動 | 半自動化（GitHub Actions 等）の検討余地はあるか |
