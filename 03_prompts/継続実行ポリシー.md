# 継続実行ポリシー（AI工場 自律実行ルール）

> Epic [#91](https://github.com/kaeru07/vault/issues/91) 確定。
> vloop / Claude Code が実行開始時に必ず参照する。
> **原則: AIは止まらず継続実行する。承認待ちを作るのは以下の5条件のみ。**

---

## 停止条件（承認待ちを作ってよい場合）

| # | 条件 | category | 例 |
|---|---|---|---|
| 1 | **Epicゴールの変更** | `goal_change` | Epicの目的・完了条件を変えようとしている |
| 2 | **課金の発生** | `billing` | 有料API導入・SaaS契約・クレジット消費 |
| 3 | **destructive変更** | `destructive` | ファイル削除・DB drop・本番データ書き換え・外部公開 |
| 4 | **収益化方針の変更** | `monetization` | approved候補のステータス変更・収益化手段の変更 |
| 5 | **複数案で結果が大きく変わる場合** | `multi_option` | 技術選定で後戻りコストが高い・方向性が分岐する |

---

## 継続してよい場合（承認待ちを作ってはいけない）

- Epicゴール達成に影響しない技術選定（ライブラリ選択・実装方式）
- リファクタリング・コード整理・コメント追加
- テスト追加・ビルド修正
- 文書化・Vault記録・レビューファイル生成
- ToDo消化（`[x]` 承認済みのもの）
- エラー修正・バグfix（破壊的でないもの）
- 既存機能の改善・UI調整（データ非破壊のもの）

---

## decisionPolicy 別の実行モード

| policy | 動作 |
|---|---|
| `autonomous` | 5停止条件以外で一切止まらない。Epic Goal達成まで自律継続 |
| `approval_required` | 重要な技術選定でも承認待ちを作る（慎重モード） |
| `budget_sensitive` | 課金リスクを特に厳しく判断する（billing条件を広く解釈） |
| `destructive_sensitive` | destructive変更を特に厳しく判断する |

---

## 承認待ちを作る場合の形式（選択式必須）

承認待ちを作る場合は **必ず選択式** で提示する。長文入力を求めない。

```
【承認待ち】<タイトル>（priority: critical|high|low）

A) <選択肢A>   <1行補足>
B) <選択肢B>   <1行補足>
C) <選択肢C>   <1行補足>（あれば）

推奨: A
理由: <1〜2行>
```

モバイルで1タップで決定できる形式を守る。

---

## 技術的停止条件（別枠・上記5条件に加えて）

vloopが止まってよい技術的な理由（ユーザー確認が必要な理由を必ず残す）:

- `git pull / commit / push` に失敗した
- 仕様矛盾・データ欠損で勝手に判断すると危険な状況
- 実行環境がない・検証不能
- Claude利用制限など物理的に継続不能
