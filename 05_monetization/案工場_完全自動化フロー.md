# 案工場 完全自動化フロー（Issue #28 統合 Runbook）

> Issue #28。**情報収集 → 案生成 → candidate → ChatGPT 承認待ち**を 1 サイクルで回す統合 Runbook。
> 各工程は #25 / #26 / #27 の既存設計に対応する**ファサード**であり新規ロジックは追加しない。
> **本ファイルは設計（Runbook）のみ。実 cron 設定 / 実スクレイパ実装 / 実 commit / 実 PR / 実 ChatGPT 承認待ち更新はしない**
> （外部アクセス・別アプリ実装は /vloop の停止条件付近 → 実装は人間承認後に該当リポジトリで）。

---

## チェーン全体図

```
[Day 0] cron 起動（VPS / 別アプリリポジトリ）
   ↓
[Step 1] 無料トレンド情報収集（#25）
   - 06_research/daily/<YYYY-MM-DD>/<source>.ndjson
   - 8 源（Google Trends / Play / App Store / Reddit / YouTube / X / PH / AI News）
   - 規約・レート順守。機密保存禁止
   ↓
[Step 2] 案 30 件生成（#26）
   - 05_monetization/idea_pool/<YYYY-MM-DD>.ndjson
   - トレンド × 収益化手段 × 動画導線 / 粗 score / dedupKey
   - status: idea（AI は idea のみ）
   ↓
[Step 3] 上位 5 件 candidate 化（#27）
   - 05_monetization/scenarios/candidate-<NNN>.md 起票
   - sortKey 降順上位 5 + 安全弁（根拠≤2 かつ 収益期待≤2 は除外）
   - 既存 candidate との核ワード突合で重複は昇格させない
   - status: candidate
   ↓
[Step 4] 妥当性評価 8 軸を埋めて pending_approval へ
   - 05_monetization/scenarios/candidate-<NNN>.md の score を 8 軸で埋める
   - 根拠 ≤ 2 は pending_approval に上げない（要追加調査）
   - status: pending_approval
   ↓
[Step 5] ChatGPT 承認待ちキューへ追記（Issue #19）
   - 20_reviews/ChatGPT承認待ち.md に 1 ブロック追記
   - chatgpt-approval-template（90_templates/）を使用
   - 承認コマンド例（`<scenarioId> approve|hold|reject`）を必ず併記（Issue #24）
   ↓
[Step 6] GitHub push
   - obsidian-vault に commit / push（[[../03_prompts/GitHub反映ルール]]）
   - commit message: `vloop: 案工場 <YYYY-MM-DD> 5件昇格`
   ↓
[Step 7] ChatGPT が方向性判断（人間チャネル）
   - `<scenarioId> approve` / `hold: <理由>` / `reject: <理由>`
   - 人間が status を確定 → [[decision-log]] 追記
   - approved のみ progress 投入 ToDo 候補へ
```

> **AI は Step 1〜6 まで。Step 7 以降は ChatGPT + 人間**。AI が approved に上げない・progress へ送らない（[[ChatGPT承認ゲート標準]]）。

## cron 設計（実装は別リポジトリで承認後）

| 項目 | 値 | 備考 |
|---|---|---|
| 実行頻度 | 1 日 1 回 | 早朝 JST（例: 06:00）。レート/規約配慮 |
| 実行ホスト | VPS（progress と同居 or 専用） | 既存 pm2 と分離が望ましい |
| 失敗時 | リトライしない・ログのみ残す | スクレイピング失敗は規約変更等の兆候 |
| 監視 | 出力ファイル数 + commit hash を progress へ POST | 1 サイクル 1 ExecutionRun |
| 停止判断 | 規約抵触 / 取得失敗連続 3 日 / 機密混入検出 | 人間判断で復旧 |

## 1 サイクルの I/O 集約

| Step | 入力 | 出力 | 失敗時 |
|---|---|---|---|
| 1 収集 | cron トリガ | `06_research/daily/<日付>/<source>.ndjson` | 該当源スキップ・ログ |
| 2 生成 | 当日 daily/ | `idea_pool/<日付>.ndjson`（30 件） | 件数不足は不足のまま記録 |
| 3 昇格 | idea_pool 当日 | `scenarios/candidate-<NNN>.md`（最大 5 件） | 安全弁にかかったものは未昇格と記録 |
| 4 評価 | candidate-<NNN>.md | 同 md（score 8 軸埋め） | 根拠≤2 は pending_approval に上げない |
| 5 承認待ち | pending_approval 化した分 | `20_reviews/ChatGPT承認待ち.md` 追記 | 追記失敗は手動補完 |
| 6 push | obsidian-vault 全変更 | git commit / push | push 失敗は再試行せず人間判断 |

## 制約（Issue #28 本文の踏襲）

- API 課金なし・無料情報源優先（#25 と同方針）
- candidate を大量生成しない（#27 上位 5 件上限）
- 上位のみ昇格（安全弁あり）
- AI による approved 化禁止
- 外部公開・課金・広告操作は /vloop 停止条件

## 未対応点 / 仮説

- 実 cron は VPS 上の別アプリリポジトリ前提（progress と分離が望ましい仮説）
- スクレイピング規約・レート制限は実装着手時に**1 源ずつ再確認**（取得可否未確認の源あり = #25 で明示）
- ChatGPT 承認待ちへの追記は Step 5 で 1 サイクル 1 追記想定。連日 5 件 × 7 日 = 35 件/週 となるため**ChatGPT 側の処理能力**がボトルネック仮説（hold/reject の自動ローテも将来課題）
