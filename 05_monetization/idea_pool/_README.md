# idea_pool 案自動生成 設計

> Issue #26。収集トレンド（[[../../06_research/トレンド収集エンジン設計]] #25）から毎回**案を30件生成**する仕組みの設計。
> **本ファイルは設計。実 30 件生成は #25 の収集出力が前提（未収集のため未生成）。**
> 案は **idea（未candidate）**。candidate 昇格は [[案プール自動昇格ルール]]（#27）・承認は人間（[[ChatGPT承認ゲート標準]]）。

## 生成の考え方（トレンド × 収益化 × 動画導線）

各 idea は3軸の掛け合わせで作る:

- **トレンド**: `06_research/daily/<日付>/summary.md` の上位語（#25 出力）
- **収益化**: [[収益化手段台帳]] の手段（アプリ/広告/動画/拡張/テンプレ/SaaS 等）
- **動画導線**: Shorts 化しやすいか（nanikiru-shorts 等の既存基盤と相性）

スコア仮置き（[[調査手段の妥当性評価]] 8軸に後段で接続。生成時は粗スコア）:
`粗score = トレンド勢い + 収益化相性 + 動画化相性 + 既存資産流用`

## 出力形式（idea_pool/YYYY-MM-DD.ndjson 1行=1案）

```
{
  "ideaId": "YYYYMMDD-NNN",
  "createdAt": "ISO8601+09:00",
  "trendSource": "google-trends|reddit|...",
  "trendTerm": "元トレンド語",
  "手段": "収益化手段台帳の行",
  "案": "1〜2文の案",
  "動画導線": "Shorts化案 or なし",
  "粗score": 0,
  "status": "idea",          // idea のみ。candidate へはAIが勝手に上げない
  "dedupKey": "正規化キー（重複除去用）"
}
```

## 30件生成ルール

- 1回の生成で**最大30件**（多すぎるノイズを抑制。30 は上限）
- トレンド源が薄い日は30未満でも可（無理に水増ししない＝未確認の数値を作らない）
- 粗score 降順で並べ、上位を idea_pool に保存

## 重複除去ルール

- `dedupKey` = 正規化（小文字化/記号除去/手段＋核ワード）でハッシュ
- 同一 dedupKey は**最新1件のみ残す**（過去 idea_pool 全体と突合）
- 既存 candidate（[[scenarios/]]）と核ワード一致するものは `status: idea` のまま「既出」注記（candidate と重複生成しない）
- 既存ファイルは削除しない（重複は新規側を抑制する方向で解決）

## 保存先構成

```
05_monetization/idea_pool/
  YYYY-MM-DD.ndjson   # その日の生成案（最大30・重複除去済）
  _README.md          # 本設計（このファイル）
  _index.md           # 粗score上位の横断ビュー（任意・後段）
```

## 完了条件への対応（正直な状態）
- 「毎回30件生成」: **設計のみ完了**。実生成は #25 のトレンド収集出力が前提で、収集が未実装のため**未生成**（"30件生成済" とは書かない）
- 「重複除去済み」: 除去ルールを定義（実行は生成時）
- commit/push: 本設計を commit/push（✅）

## 安全弁（重要）
- 生成物は **`status: idea` のみ**。AI は idea→candidate→pending_approval→approved を勝手に上げない
  （昇格は [[案プール自動昇格ルール]] #27＋承認ゲート [[ChatGPT承認ゲート標準]]）
- 外部公開・課金・広告に踏み込む案でも、生成は記録のみ（実行しない）

## 未対応点 / 仮説
- 実生成は #25 収集が動いてから（順序依存）。本タスクは設計まで（停止条件の別アプリ実装/外部アクセス回避）
- 粗score の重みは初期仮説。#27 で candidate 昇格時に妥当性評価8軸へ接続
- idea_pool の寿命管理（古い idea の間引き）は #27 と合わせて別途定義
