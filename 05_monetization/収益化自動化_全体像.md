# 収益化自動化 全体像

> 収益化に向けた動きを **調査→候補化→妥当性評価→人間承認→承認済みのみ実行→Vault蓄積** の一方向パイプラインで回す。
> **AIは勝手に案件を増やして作業開始しない。** 承認は必ず人間。

## 1. 収益化パイプライン全体図

```
[調査] Scrape Lab / ChatGPT / Claude / 手動
   ↓ 調査レポート（妥当性評価付き）→ Vault 06_research・05_monetization
[候補化] 収益化手段台帳 + シナリオテンプレで candidate 化
   ↓
[妥当性評価] 調査手段の妥当性評価.md のスコア付け（自動着手しない）
   ↓
[人間承認] 収益化シナリオ承認フロー.md（candidate→…→approved は人間のみ）
   ↓ approved だけ
[progress連携] progress連携基準.md の条件を満たすもののみ progress へ
   ↓
[実行] Claude Code / Codex が承認済み作業を実行（既存集中作業モード）
   ↓
[結果蓄積] ExecutionRun(progress 正本) + 20_reviews + Vault(判断・長期記憶)
```

最優先判断軸は常に **収益化インパクト**。スコア合計が高くても自動着手はしない（必ず承認待ち）。

## 2. 役割分担

| 主体 | 役割 | できること | できないこと |
|---|---|---|---|
| **Vault（GitHub管理）** | 調査・候補・承認の**正本** | 設計/台帳/承認状態の記録 | 自動実行 |
| **progress** | 承認済み作業の実行管理 | queue/ExecutionRun/可視化 | 未承認の着手 |
| **Claude Code** | 調査補助・承認済み実装 | 調査、Markdown整備、approved作業 | candidate→approved 昇格、勝手な案件追加 |
| **Codex** | 読み取り専用レビュー・対立確認 | レビュー要約、改善ToDo提案 | 承認、本体破壊 |
| **Scrape Lab** | 市場/競合データ収集 | 規約順守の範囲で収集 | 過剰取得、承認 |
| **ChatGPT** | 構想出し・レビュー・台帳更新案 | 案出し、レビュー、台帳ドラフト | GitHubへ直接approved化 |
| **人間（あなた）** | **承認者** | candidate→approved の判断、撤退判断 | — |

## 3. 入口から出口までの流れ

1. **入口**: 調査トリガ（定期調査運用.md の段階に従う）
2. 調査結果に妥当性評価（調査手段の妥当性評価.md）を必ず付与
3. 収益化手段台帳.md の手段に紐付けて candidate 登録（シナリオテンプレ使用）
4. スコアリング → 並べる（ただし**自動着手禁止**・pending_approval へ）
5. 人間が承認フローで approved/rejected/hold を決定
6. approved かつ progress連携基準.md を満たすものだけ progress へ1作業に分解して送る
7. Claude Code/Codex が実行 → ExecutionRun + 20_reviews
8. 結果を Vault に蓄積し、次サイクルの調査・評価にフィードバック

## 4. AIが自動でできること / 人間が承認すること

**AIが自動でできる（承認不要）**
- 調査の実行と調査レポート作成（妥当性評価付き）
- candidate の起票（pending_approval 状態で）
- スコアリングと並び替え（提案まで）
- approved 済み作業の実行・検証・記録

**人間が必ず承認する（AI禁止）**
- candidate → approved への昇格
- progress への送り込み可否
- 収益化手段の新規追加で大きな投資/公開を伴うもの
- 撤退・凍結判断
