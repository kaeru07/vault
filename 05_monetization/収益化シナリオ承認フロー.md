# 収益化シナリオ承認フロー

> 収益化シナリオのステータス遷移を定義。**AIは approved へ勝手に移動しない。承認済みだけ progress へ送る。**

## ステータス定義

| status | 意味 | 遷移できる主体 |
|---|---|---|
| `candidate` | 調査から起票された候補。未評価/評価中 | AI可（起票） |
| `pending_approval` | 妥当性評価済み。人間承認待ち | AI可（評価完了で昇格） |
| `approved` | 人間が承認。progress送り対象 | **人間のみ** |
| `rejected` | 不採用 | **人間のみ** |
| `hold` | 保留（情報不足/時期尚早） | 人間のみ（AIは hold 提案まで） |
| `in_progress` | progress側で作業中 | progress運用（approved起点） |
| `published` | 公開/リリース済み | 人間確認後 |
| `reviewing` | 公開後レビュー/改善検討中 | AI可（レビュー）/人間（判断） |

## 遷移ルール（厳守）

```
candidate ──(AI: 妥当性評価完了)──▶ pending_approval
pending_approval ──(人間のみ)──▶ approved / rejected / hold
approved ──(progress連携基準を満たす)──▶ in_progress
in_progress ──▶ published ──▶ reviewing
hold ──(人間)──▶ pending_approval（再評価）
reviewing ──(人間)──▶ approved（改善着手）/ rejected（撤退）
```

**禁止事項（AI）**
- `pending_approval → approved` をAIが行う（**絶対禁止**）
- `candidate` を飛ばして直接 progress へ送る
- スコアが高いことを理由に自動着手する
- 人間承認なしに公開・課金・外部公開を伴う作業を開始する

## 承認時に人間が確認する最小項目
- 収益化インパクト（最優先軸）
- 調査根拠の強さ（一次情報か）
- 対象アプリ/手段が明確か
- 最初の作業が1つに分解できるか
- 完了条件があるか

承認は本ファイルまたはシナリオファイルの `status:` を人間が書き換えることで成立。AIは差分提案のみ。
