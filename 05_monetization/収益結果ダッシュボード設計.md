# 収益結果ダッシュボード設計

> Issue #9。**収益結果（実績指標）**を見える化し、どの施策が効いたか判断できるようにする。
> [[progress_収益パイプライン設計]]（#4=プロセス進行の可視化）とは別レイヤ＝**結果（数値）の可視化**。
> **実装しない。設計のみ。** 数値は事実のみ・推測は推測と明記する原則を画面にも適用。

## 1. 指標一覧

| 指標 | 定義 | 粒度 | 収益化インパクト軸との対応 |
|---|---|---|---|
| DL数 | アプリDL累計/月次増分 | アプリ別 | 収益期待の先行指標 |
| 動画再生数 | Shorts 再生/視聴維持 | チャンネル/動画別 | 動画化相性・送客 |
| 広告収益 | AdMob/AdSense 等の確定収益 | アプリ/サイト別・月次 | 収益期待（実績） |
| SNS流入 | 流入元別セッション | 媒体別 | 拡散性・送客効果 |
| 成功率 | published 到達 / approved 件数 | パイプライン全体 | 承認→公開の歩留まり |
| 月次推移 | 上記の前月比 | 月次 | 方向修正の判断材料 |

## 2. データ元（取得元と種別を明示・推測禁止）

| 指標 | データ元 | 種別 | 取得方法（当面） |
|---|---|---|---|
| DL数 | Google Play / App Store Console | 公式一次 | 人手で月次転記（API/課金なし方針） |
| 動画再生数 | YouTube Studio | 公式一次 | 人手で月次転記 |
| 広告収益 | AdMob / AdSense 管理画面 | 公式一次 | 人手で月次転記 |
| SNS流入 | 各SNS/アクセス解析 | 公式/二次 | 人手で月次転記 |
| 成功率 | Vault シナリオ status + decision-log | 内部一次 | 集計（[[収益化ステップ一覧]]） |
| 月次推移 | [[../20_reviews/月次収益レビュー]] | 内部一次 | 月次レビューから集約 |

- **未確認の数値は「未確認」と表示**（確認済みと書かない／推測は推測と明記）
- 外部 API / 外部 DB は使わない（既存方針踏襲）。当面は月次手入力 JSON

## 3. 画面案（実装は別ToDo・承認後）

```
┌ 収益結果ダッシュボード ──────────── 月: 2026-05 ▼ ┐
│ 実収益(確定) ¥0   DL累計 —   Shorts再生 —   成功率 0/0 │
├──────────────┬───────────────────────┤
│ 月次推移(折れ線)        │ 施策別効果(表)                  │
│ 収益/DL/再生 の前月比   │ candidate-XXX → 指標変化 → 効/不効  │
├──────────────┴───────────────────────┤
│ データ鮮度: 各指標の最終更新日 / 未確認は ⚠ 表示           │
└──────────────────────────────────────┘
```

- データ供給: `progress` ローカル `revenue-metrics.json`（月次手入力・ExecutionRun と疎結合）
- [[progress_収益パイプライン設計]] と相互リンク（プロセス↔結果を行き来）
- 収益額・成功率は確定値のみ太字。未確定は薄字＋⚠

## 4. データ構造（案・実装しない）

```
RevenueMetric {
  month: "YYYY-MM"
  app: string
  dl?: number            // 未取得は null（推測値を入れない）
  videoViews?: number
  adRevenue?: number     // 確定値のみ。通貨明記
  snsInflow?: number
  source: "公式一次|内部一次|未確認"
  updatedAt: string
}
```

## 未対応点 / 仮説
- 自動取得は API/課金なし方針のため当面**月次手入力**（自動化は Phase 後送り）。仮説: 公開アプリが出てから着手で十分
- 成功率の分母（approved 件数）は現状 0 のため指標として動くのは承認後
- progress 実装は本番稼働アプリ＝設計のみ／実装は人間承認後（[[../03_prompts/Claude-Code標準運用]]）
