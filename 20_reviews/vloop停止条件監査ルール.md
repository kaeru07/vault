# vloop 停止条件監査ルール

> Issue #66 対応。vloop / vrun が「全 ToDo を消化する前に早期終了していないか」を監査し、**停止理由の正当性を判定**するためのルール。
> 停止条件の正本は [[../03_prompts/claude-commands]]（vloop コマンド本文）。本ページはそれを監査可能な形に固定する。

## 0. 背景

- 旧運用で vloop が「安全にここまで」で早期終了し、Epic 完了前に止まる事故があった（Issue #50 / #66）。
- 「コメントだけ付けて完了扱い」「次の小 Issue を作っただけで停止」などの**不正な停止**を構造的に検出するため、停止理由を分類・判定する基準を明文化する。

## 1. 正当な停止理由（このいずれかなら停止してよい）

1. Epic の完了条件を満たした
2. 破壊的変更が必要（rm / 大量 mv / スキーマ破壊 等）
3. 認証情報・課金・外部公開など人間判断が必要
4. 明確な規約違反リスクがある
5. 実行環境がなく物理的に進められない（対象アプリ未 clone / サービス停止 等）
6. git pull / commit / push が失敗した
7. Claude Code の利用制限が近い
8. 収益化 candidate が approved でなく、approved 必須の作業しか残っていない（[[vloop_queue]] が空）

## 2. 不正な停止理由（これで止まったら早期終了 = 違反）

- 小さい Phase が 1 つ終わっただけ
- 追加調査が残っているのに止めた
- candidate 化判断が残っているのに止めた
- レビュー用まとめが未作成
- 既存設計へのリンク確認だけで止めた
- 「次の Issue を作ればよい」と判断しただけ
- **コメントだけ付けて done 扱い**にした
- 「完了っぽい」と曖昧判断で止めた
- レビュー対象を明記せず終了
- 未処理 Issue を一覧化せず終了

## 3. 停止理由の正当性判定（Step 9 で必ず実施）

vloop / vrun 終了時に、以下を 1 行で出す:

```
停止理由: <理由>
正当性判定: 正当 / 不正
根拠: §1 のどれに該当するか（不正なら §2 のどれに該当したか）
```

- §1 のいずれかに**明確に**該当 → 正当
- §1 に該当せず §2 に当てはまる → 不正（= 継続すべきだった）。次サイクルで継続する。
- 判定に迷う場合は「継続できる仮説があるか」を問う。仮説を置いて進められるなら**継続が原則**（止まらない側に倒す）。

## 4. 早期終了の事後監査（過去ログ点検）

- `03_prompts/claude-commands/logs/vloop_*.md` の「停止理由」「次の一手」を時系列で点検する。
- 「次の一手」に実行可能な作業が書かれているのに次サイクルで着手されていない場合 = 取りこぼし。次サイクル優先キューへ。
- 同じ Epic が複数ログにまたがって「途中で止まった」状態が続く場合 = 早期終了の疑い。Epic 完了優先で一気に進める。
