# Vault蓄積運用ルール

#運用ルール #Vault #Obsidian #レビュー #ToDo

> [!important]
> 今後、ユーザーが確認する正本は GitHub Issue ではなく、Vault 内のノートにする。
> Issue は Claude/vloop 側の内部管理・作業実行用として扱い、ユーザー確認用の情報は Vault に蓄積する。

---

## このルールの目的

これまで、ToDo・レビュー・承認待ち・進捗確認が GitHub Issue 側に寄りすぎていた。
しかし、ユーザーは Issue を日常的に見ていない。

そのため今後は、以下のように役割を分ける。

| 役割 | 置き場所 |
|---|---|
| ユーザーが見る情報 | Vault 内ノート |
| iPhone Obsidianで確認する入口 | `00_START_HERE.md` |
| 案・候補・判断材料 | Vault 内の案/承認ページ |
| レビュー結果 | Vault 内のレビュー記録 |
| Claude/vloop の内部作業管理 | GitHub Issue |

---

## 基本方針

> [!summary]
> Issue は裏側の作業チケット。
> Vault はユーザーが読む正本。

今後、ChatGPT または Claude/vloop が作業した場合、Issue だけに記録して終わらせない。
必ず、ユーザーが見るべき内容を Vault 側にも残す。

---

## Vault に必ず残すもの

### 1. ToDo を追加したとき

Issue を作っただけで終わらせない。
以下のどこかに、ユーザー向けの要約を残す。

- `20_reviews/次に実体化するToDo.md`
- `05_monetization/epics.md`
- `00_START_HERE.md`
- 関連する案ページ

残す内容:

- 何をやるのか
- なぜ必要なのか
- 今どの状態か
- 次に Claude/vloop が何をすべきか
- ユーザー確認が必要か

---

### 2. レビューしたとき

チャット上だけでレビュー完了にしない。
Vault にレビュー結果を残す。

候補ファイル:

- `20_reviews/レビュー記録.md`
- `20_reviews/Issue完了判定ルール.md`
- `20_reviews/次に実体化するToDo.md`
- 対象案の個別ノート

残す内容:

- レビュー対象
- 作業状態: 完了 / あなたの確認待ち / 未対応 / 統合済み / 停止中
- レビュー状態: レビュー済み / 追加対応あり / 未レビュー
- 何が変わったか
- できるようになったこと
- まだ残っていること
- 次に見る場所

---

### 3. 承認待ちが発生したとき

Issue ではなく、Vault の承認ページへまとめる。

候補ファイル:

- `20_reviews/ChatGPT承認待ち.md`
- `05_monetization/案の情報源と採用理由.md`
- `05_monetization/候補一覧.md`

残す内容:

- 何の案か
- なぜ候補なのか
- 情報源
- API代が必要か
- 収益化導線
- リスク
- 承認したら何を作るか
- 保留/却下する理由

---

### 4. 案を作ったとき

案そのものだけでなく、なぜその案になったかを残す。

残す流れ:

```text
情報源
↓
気づき
↓
案
↓
候補化理由
↓
試作状況
↓
判断
```

主な保存先:

- `05_monetization/案の情報源と採用理由.md`

## Issue と Vault の関係

| 項目 | GitHub Issue | Vault |
|---|---|---|
| Claude/vloop の作業指示 | 使う | 要約だけ残す |
| ユーザーが日常的に見る | 見ない | 見る |
| 承認判断 | 裏側の参照 | 正本 |
| レビュー結果 | コメント可 | 必ず残す |
| 進捗の現在地 | 補助 | 正本 |

---

## 今後のレビュー時ルール

ユーザーが「レビューして」と言ったら、以下を行う。

1. Issue や作業報告を確認する
2. チャットで状況連携する
3. 必要な場合は Vault にレビュー結果を追記する
4. Issue だけに記録して終わらせない

レビュー時のチャット返答には必ず以下を含める。

- 何が変わったか
- できるようになったこと
- まだ残っていること
- ユーザー確認が必要なこと
- Vault のどこを見ればよいか
- 次に実行するもの

---

## 今後のToDo追加時ルール

ユーザーが「ToDo追加して」と言ったら、以下を行う。

1. 必要なら Issue を作る
2. ただし Issue だけで終わらせない
3. Vault の該当ページにも反映する
4. 実体未作成なら `planned_only` と明記する
5. 次サイクルで実体化する順番に入れる

| 4 | `20_reviews/ChatGPT承認待ち.md` | ユーザーが承認/保留/却下する入口 |
| 5 | `00_START_HERE.md` | iPhone Obsidianの最初の入口 |

---

## 禁止事項

- Issue だけ作って「ユーザーが確認できる」と扱わない
- Issue コメントだけでレビュー完了扱いしない
- 実体ページがないのに「ここを見れば分かる」と案内しない
- `planned_only` と `done` を混同しない
- ユーザー確認待ちを完了扱いしない
