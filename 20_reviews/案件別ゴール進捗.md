# 案件別ゴール進捗

>
> vloop の実行対象は [[vloop_queue]]、全ToDoの台帳は [[案件別ToDo一覧]]。

## 見方

- **ゴール**: その案件で最終的に達成したい状態
- **現在地**: 今どこまで来ているか
- **詰まり**: 何が原因で前に進みにくいか
- **改善するToDo**: どの vloop_queue のToDoが終われば進むか
- **ユーザー判断**: あなたのチェックが必要なもの

# 1. Vault運用

## ゴール

- IssueはChatGPTからToDo追加する入口として残す
- ユーザーが見る正本はVaultにする
- vloopは [[vloop_queue]] の `[x]` だけを自動実行する
- `[ ]` は承認待ちとして実行しない
- 作業途中の通常確認は不要にし、停止条件に当たるまで進める

## 現在地

- [[vloop_queue]] は作成済み
- `[x]` はvloop対象、`[ ]` は承認待ちのルールを追加済み
- 案件別ToDo一覧は正本として整備済み
- ただし、実際に各案件の進捗とvloop_queueの紐づけはこのページで整備中

## 詰まり

- どのToDoが終わればどの状況が改善するかが見えづらかった
- START_HERE / 案件別ToDo一覧 / vloop_queue / 完了ログの役割が混ざりやすい

## 改善するToDo

| vloop_queue のToDo | 終わると改善すること |
|---|---|
| 案件別ToDo一覧を正本として維持し、Issueだけに残っている新規ToDoをVaultへ反映する | Issueだけで進む問題が減る |
| 完了ToDoを `20_reviews/完了ToDoログ.md` へ移動する | 案件別ToDo一覧が見やすくなる |
| `00_START_HERE.md` から `vloop_queue.md` と `案件別ToDo一覧.md` へ辿れるようにする | iPhoneで迷いにくくなる |
| `20_reviews/次に実体化するToDo.md` を正本扱いから外し、非推奨/参照用にする | 二重管理が減る |
| 重複ToDoを目的ベースで統合し、統合元は merged 扱いにする | 同じ目的のToDo増殖が減る |

## ユーザー判断

- [ ] 旧運用ファイルを削除する

# 3. トークン速度・LLMベンチ

## ゴール

- 作る価値があるか、低コストで成立するかを判断する

## 現在地

- APIなし方針、手入力、ログ貼り付け、JSON保存の案はある
- ただし、専用司令塔ページが未整備で、現在地が追いづらい
- 実際の最小試作は承認待ち

## 詰まり

- APIなしでどこまで成立するかの仕様が見えづらい
- 実試作へ進むかはユーザー判断待ち

## 改善するToDo

| APIなしでできる範囲、手入力/ログ貼り付け/JSON保存の方針を整理する | 作る価値の判断材料が増える |
