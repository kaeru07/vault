# レビュー状態ラベル運用

> Issue #65 対応。レビューが完了したかどうかを **GitHub Issue ラベル**で一目で分かるようにするための運用ルール。
> 状態分類そのものの定義は [[Issue完了判定ルール]] §2 を正本とし、本ページはその **GitHub ラベル実装と適用手順**を定める。

## 0. 背景

- [[Issue完了判定ルール]] §2 でレビュー状態（reviewed_ok / reviewed_followup / user_check / not_reviewed）は定義済みだったが、**GitHub ラベルとしては未作成**で運用が回っていなかった（#65 コメント 2026-05-23 user_check）。
- 本ルールでラベルを確定し、vloop / vrun のレビュー結果を Issue ラベルへ反映する運用へ切り替える。

## 1. ラベル体系（確定）

| GitHub ラベル | 意味 | 色 |
|---|---|---|
| `review:ok` | レビュー済み・問題なし | green |
| `review:followup` | レビュー済み・追加対応あり | yellow |
| `review:user_check` | ユーザーレビュー待ち（Claude 実装済み・人の確認待ち） | orange |
| `review:not_reviewed` | 未レビュー | gray |

- `review:` 名前空間を付け、既存のトピックタグ `review`（黄）と区別する。
- レビュー状態は **常にこの 4 つのうち 1 つだけ**を Issue に付与する（排他）。
- Issue の**完了状態**（done / open / blocked 等）は別軸。状態分類は [[Issue完了判定ルール]] §1、レビュー状態は本ラベルで表す。

## 2. 適用ルール

- **新規 Issue**: ラベルなし（= 暗黙の `review:not_reviewed`）。明示したい場合のみ `review:not_reviewed` を付ける。
- **Claude が実装・対応した Issue**: 自己検証まで済ませたら `review:user_check` を付ける（人の最終確認を待つ意味）。Claude が単独で `review:ok` を付けてよいのは、完了条件が**機械的に検証可能**で実際に検証した場合のみ。
- **ChatGPT / ユーザーがレビュー済み**: `review:ok` または `review:followup` へ付け替える。
- **承認ゲート対象（収益化 candidate）**: レビュー状態とは別に approved 判断が必要。レビュー状態ラベルだけでは approved にならない（[[vloop_queue]] 参照）。

## 3. ラベルの付け替え

- 状態が変わったら**古いラベルを外して新しいラベルを付ける**（重複させない）。
- 付け替えコマンド例: `gh issue edit NN --add-label "review:ok" --remove-label "review:user_check"`

## 4. vloop / vrun との連携

- vloop / vrun の Step 9（[[Issue完了判定ルール]] §4）で全 Issue を状態分類した後、対応した Issue に本ラベルを反映する。
- 「コメントだけ付けて終わる」のは禁止。状態分類コメント + レビュー状態ラベルをセットで残す。
